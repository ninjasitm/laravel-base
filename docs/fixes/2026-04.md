## [2026-04-05] - Tagging Job Fails on Detached HEAD and Masked Push Errors

### Fixed/Changed/Added

- **Issue**: The release tag phase referenced `master` directly, failed in detached HEAD CI checkouts, and still reported success even when push operations failed.
  - **Root cause**: `bump-git-tag.sh` hard-coded `master`, did not enable fail-fast shell behavior, and performed a second push via `origin` (`git push --tags`) that can fail with CI token permissions.
  - **Solution**: Updated the script to resolve target commit from `${CI_COMMIT_SHA:-HEAD}`, enable `set -euo pipefail`, detect existing tags using `git tag --points-at`, and push only the created tag through `tag-origin` when configured (with explicit CI env validation) or `origin` fallback outside CI.
  - **Files modified**: `bump-git-tag.sh`, `tests/Unit/BumpGitTagScriptTest.php`
  - **Status**: ✅ Fixed

### Impact

- 🔧 **Technical**: Tagging now works with detached HEAD checkouts and fails immediately on command/push/auth errors instead of masking failures.
- 🎯 **User Experience**: CI release logs are now trustworthy, reducing false-positive “success” outcomes.
- 🚨 **Breaking Changes**: None.

## Patterns & Learnings

### Common Issues Discovered

- Release scripts that rely on branch names can break in CI because many pipelines run in detached HEAD mode.

### Best Practices Established

- Use `${CI_COMMIT_SHA:-HEAD}` in CI-aware scripts instead of hard-coded branch refs.
- Use `set -euo pipefail` for release scripts so job status reflects actual failures.
